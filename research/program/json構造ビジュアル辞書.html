JSONの入れ子構造 ビジュアル辞書

枠の色 = 深さ（ネストのレベル）／文字の色 = データの種類

① まず種類の見分け方

🔑 キー（項目名・必ず文字列） 📝 "文字列" 🔢 数値 ✅ true/false 🚫 null（値なし）
"名前": "ゲゲゲ", "レベル": 3, "討伐済み": false
{ } の中に「キー: 値」がコロンで並んでるのが object（オブジェクト）。
キーは絶対に文字列やけど、値はどの種類でもOK。

② { } と [ ] の違い

object { } = 「名前付きの箱」
"名前": "アズキ"
"種類": "犬"
array [ ] = 「順番に並んだ箱」 "タスク1" "タスク2" "タスク3"
object は「名前で取り出す」（"名前"って聞いたら"アズキ"が出る）。
array は「順番で取り出す」（1番目・2番目...で管理、名前はない）。

③ 入れ子（ネスト）＝箱の中に箱

深さ0：一番外側
"妖怪": ↓
深さ1：妖怪オブジェクト
"名前": "締切妖"
"タグ": ↓
深さ2：タグの配列 "緊急" "仕事"
"ルーティン": ↓
深さ2：ルーティン情報
"周期": "毎日"
"次回": "2026-07-05"
枠の色が変わるたびに「1段深くなった」って意味。
外の箱の中に、また { } や [ ] が入ってるのが入れ子。何段でも重ねられる。
▶ これを生のJSONで見るとこうなる
{ "妖怪": { "名前": "締切妖", "タグ": ["緊急", "仕事"], "ルーティン": { "周期": "毎日", "次回": "2026-07-05" } } }

④ 配列の中にオブジェクトが並ぶパターン（一番よく使う）

深さ0："タスク一覧"の配列
配列の1番目
"id": 1
"タイトル": "買い物"
"完了": false
配列の2番目
"id": 2
"タイトル": "通院"
"完了": true
これが「タスクを配列で持って、1件ずつがオブジェクト」の形。
妖怪プランナーみたいに「同じ形のデータがたくさん並ぶ」場合は、だいたいこのパターンになる。

⑤ 迷子にならんためのコツ

・{ } の数と [ ] の数は必ずペア（開いたら必ず閉じる）

・エディタで見るとき、閉じ括弧がどのキーに対応してるか分からんくなったら → 上のような色付き枠を頭の中で想像する

・「このカンマは何と何を区切ってる？」がわかれば、だいたい構造は読める

・深さが3段以上になってきたら、そもそも構造を分割した方がええサインかも

⑥ 書き方の基本ルール

①キーは必ず"ダブルクォート"で囲む（シングルクォート不可）

"名前": "OK"
'名前': "NG（シングルクォートは無効）"

②最後の項目の後にカンマを付けない（末尾カンマ＝トレイリングカンマはエラー）

"a": 1, "b": 2, ← これがNG（bの後にカンマ不要）

③コメントは書けない（// メモ みたいなのはJSONでは無効。JSに埋め込む時だけコード側でコメント可）

④文字列の中で特殊文字を使う時はバックスラッシュでエスケープする

"セリフ": "彼は\"やめとけ\"と言った"
"改行入り": "1行目\n2行目"

⑤数値には引用符を付けない（つけると文字列になり計算できなくなる）

"レベル": 3 ← OK（数値として扱われる）
"レベル": "3" ← 一応動くけど文字列扱いなので計算に使えない

⑥キーの命名は好きに決められる（JSON自体に決まりはない）が、プロジェクト内で統一するのが大事：キャメルケースtaskName かスネークケースtask_name、どちらかに揃える

JSONファイル単体は「データを表すためだけの形式」。
ロジック（if文とか計算）は書けへんから、変わりうる値はJSON、書き方・ロジックはコード側、という分担になる。
