RES-008 ・ ポケカAIアドバイザー

OCR共通スキーマ ロジック図

2026-07-21 更新版（設計確定・実装はまだ）

2026-07-21の更新内容：subtypes／ruleTagsの導出ルールを1項目ずつ見直した。ACE SPECはポケモンには付かないラベルなので、ruleTags側の導出元をsubtypesから外し、下の「横断フラグ（cross_category_fields）」という別枠に移した（トレーナーズ・エネにも共通で付き得るため）。VMAXはレギュ落ち済みで、値は保持するが特別扱いはせずregulationMark判定に任せる方針に確定。hasAbility（特性の有無）はsubtypesにはあるが、deckRules（枚数ルール）には一切関与しないため「今は繋がない・将来のAI相談機能用の情報フラグ」と明記した。
INPUT
カード生データ（TCGdex／OCR結果）
カード名・印刷されてる属性・わざ欄・進化段階表記・ex/メガ等のラベル表記・特性欄の有無など、そのまま読み取れる情報一式
↓ そのまま保持する情報と、分類ラベルとして抜き出す情報に分かれる
そのまま保持（そのまま入れる、分岐なし）
type / moves[].cost
・type：ポケモン自身の属性（1個）
・moves[].cost：技ごとの消費エネルギー（技の性質で複数属性持つ可能性あり）
例：マシマシラはtype「エスパー」だがmoves[].costには「悪」が入る。属性と技コストは常に別フィールド
分類ラベルとして抽出（ポケモン専用）
subtypes
・進化段階（たね/1進化/2進化）
・特殊ラベル系（ex/メガ/VMAX/テラスタル/未来/古代/かがやく）
・どうぐ扱いポケモンか
・特性の有無（hasAbility）
ACE SPECはここには含まない（ポケモンに付かないラベルのため。下の「横断フラグ」枠を参照）。VMAXはレギュ落ち済みだが値としては保持し、除外はregulationMark判定に任せる。hasAbilityはsubtypesとしては持つが、ruleTags側へは導出しない（右下の注記参照）
type / moves[].cost はここで一旦横に置いておく（deckRulesとは直接つながらない）
↓ subtypes（どういうポケモンの分類か）を見て、性質フラグ（各々の処理用フラグ）を導出する
DERIVE（導出）
ruleTags
・exラベル → 「サイド取られる枚数変化」フラグ
・未来/古代/かがやく等 → 「レギュレーションで有効/無効が変わる」フラグ（実際の合否判定は既存のregulationMark照合が担当）
・どうぐ扱いポケモン → 「ポケモンだがトレーナーズ扱い」等の制約フラグ
ruleTagsはフラグを持つだけ。「何枚まで」「合法かどうか」の実計算はここではしない。ACE SPECはポケモンのsubtypesに存在しないため、ここでの導出対象からは外れる（下の横断フラグ枠を参照）。hasAbilityも意図的にここへは導出しない（2026-07-21確定）
↓ ここでポケモン以外（グッズ／どうぐ／サポート／スタジアム／エネ）のカードが合流する
CROSS-CATEGORY（横断フラグ）
cross_category_fields
・isAceSpec：true/false（グッズ・どうぐ・サポート・スタジアム・エネのいずれにも付き得る）
・isBasicEnergy：true/false（エネのみ対象）
deckRulesの枚数判定に直接使う2フラグのみを構造化。「相手のグッズを使えなくする」「どうぐを破壊する」「ベンチ最大数を増やすスタジアム」等の対戦中の効果は、デッキ登録時の枚数には影響しないため構造化対象外（trainer_card_rulesで対戦中ルールとして別管理、原文テキスト保持）
↓ ruleTagsとcross_category_fields、両方を材料に構築枚数等のルールを計算する
CALCULATE（計算）
deckRules
・同名カード基本4枚まで（名前基準の合算）
・isAceSpec:trueのカードは合計1枚まで（cross_category_fields参照）
・isBasicEnergy:trueのカードは無制限（cross_category_fields参照）
・デッキ合計60枚ちょうど
・ポケモン最低1枚必須（事故率の話は?ask機能側、ideas.md今後検討に記録済み）
↓
OUTPUT
デッキ構築 可否判定結果
枚数ルール違反があれば一覧化して返す（既存のレギュ判定violations方式と同じ考え方を踏襲予定）
読み方：上から下へ「OCR生データ→分類ラベル→性質フラグ→計算ルール→判定結果」の順で処理が流れる想定。type/moves[].costだけは枚数ルールに関与しないため本流から外れて横に置いてある（将来の対戦シミュレーション機能で技コストを参照する可能性はideas.mdに記録済み）。

subtypes／ruleTagsはポケモン専用の分類・導出フロー。グッズ／どうぐ／サポート／スタジアム／エネは別カテゴリのため、この2層は通らず、横断フラグ（cross_category_fields）としてdeckRulesに直接合流する。トレーナーズの対戦中の使用ルール（1ターン1枚・場に1枚まで等）はこの図の範囲外で、trainer_card_rulesセクションが別途管理する。
