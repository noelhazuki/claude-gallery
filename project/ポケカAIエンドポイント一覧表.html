RES-008 / pokeca-ai

エンドポイント一覧表

Worker（pokeca-ai-worker）が持つ全エンドポイントの一覧。「別Workerを新規に作らず、既存の1つのWorkerにURLパラメータでの分岐を増やす方式」のため、全て同一Workerへの?〇〇=true形式のクエリパラメータ呼び出し。
実装済み 未着手・計画中 不採用・廃止済み
① meta系（環境デッキ：deck:meta:{id}）
メタデッキは基本的に直接フル編集しない方針。中身を直したい時はcopy_mineでmineへ複製してから編集する運用。
?register_meta=true実装済み
環境デッキを新規登録する
methodPOST
必須項目id / name / cardList / howToPlay
備考同一id既存時は409エラー。cardListはresolve_cardlistの出力を人が確認UIでチェックしてから渡す想定
?delete_meta=true実装済み
環境デッキを削除する
?copy_meta=true実装済み
環境デッキを複製する
備考sourceIdから元デッキを取得し、新id・新名称を自動生成して複製
?update_meta_howtoplay=true実装済み（2026-07-19）
howToPlay（回し方メモ）だけを更新する
備考name / cardList / deckCodeは更新不可。「フル編集はしないがメモだけ書き足したい」要望に対応
?get_meta_howtoplay={id}実装済み（2026-07-19）
howToPlay単体を取得する
備考list_metaが軽量レスポンス（id/nameのみ）でhowToPlayを含まないため、編集パネル表示専用に追加
?list_meta=true実装済み
環境デッキの一覧を取得する
備考id/nameのみの軽量レスポンス（howToPlay等は含まない）
?update_meta=true（フル編集）不採用・実装しない方針（2026-07-19確定）
cardList等を含む全項目編集
理由メタデッキは直接編集せず、copy_mine（sourceType:"meta"）でmineへコピーしてから編集する運用に統一したため
② mine系（自分のデッキ：deck:mine:{id}）
?register_mine=true実装済み
自分のデッキを新規登録する
methodPOST
必須項目id / name / cardList
任意項目concern
?update_mine=true実装済み
自分のデッキを更新する（マージロジック込み）
備考locked:trueのレコードは拒否される（誤上書き防止）。ロックボタンからは{id, locked:true}のみ送信される
?unlock_mine=true実装済み
ロックを解除する
備考自動ロック解除の経路はなく、常にこのエンドポイントの明示的な呼び出しでのみ解除される
?delete_mine=true実装済み
自分のデッキを削除する
備考deck:mine:backup:{id}（1世代前スナップショット）も同時に削除される
?copy_mine=true実装済み
デッキを複製する（Windows方式の自動命名）
備考sourceType:"meta"／"mine"どちらも指定可。meta→mineコピー時はhowToPlayを「[元メタの回し方] 〜」形式でconcernへ引き継ぐ。複製後は常にlocked:falseにリセット
?recheck_mine=true実装済み（2026-07-20）
mine1件分のprovisionalカードをまとめて再照合する
備考ヒット分はconfirmedへ昇格。ロック中でも実行可（枚数・構成は変えず正体確定のみのため）。setInfo/setCodeを持たない旧provisionalはskipped扱い。awaitStatus:'manual'のエントリーは再照合対象から自動で除外される（manualSkippedとしてカウント）
?set_manual=true実装済み（2026-07-21）
provisional1件のawaitStatusを手動で'manual'へ切り替える
methodPOST
必須項目deckId / category　＋　（pid）または（tempName / setInfo）
備考対象特定はdeckId＋category＋tempName＋setInfoの完全一致（findProvisionalEntry関数）。setInfoはポケモンならsetInfo値、トレーナーズ／エネならsetCode値を同じ欄に渡す。recheck_mineと同じ理屈でロック中のデッキでも実行可。切替後はrecheck_mineの再照合対象から外れる
2026-07-21追記：pid（背番号）を任意項目として追加。pidを渡す場合はtempName/setInfoは省略可（categoryのみ引き続き必須）。pidを渡さない場合は従来通りtempName/setInfo必須（pid実装前の古いprovisional向けの経路として存続）
?update_wait_days=true実装済み（2026-07-21）
provisional1件のwaitDaysを更新する
methodPOST
必須項目deckId / category / waitDays　＋　（pid）または（tempName / setInfo）
備考対象特定はset_manualと同じくfindProvisionalEntry（pid優先、無ければtempName/setInfo）。waitDaysは絶対値方式（UIの「+7」ボタン相当の計算はフロント側で行い、ここには計算済みの絶対値を送る）。1以上の整数のみ許可。ロック中でも実行可
?list_mine=true実装済み
自分のデッキの一覧を取得する
備考全データを返却（list_metaと違い軽量化していない）
③ カード取得・照合の基盤
?get_card={cardId}実装済み
カード表示用データ（名前・画像・技等）を取得する
方式遅延キャッシュ（card:{cardId}確認→無ければTCGdexから取得して保存）
備考見つからない場合は404
?resolve_card_id=xxx実装済み
setInfo（公式カード番号表記）→cardId の単体変換
用途デバッグ用
?resolve_cardlist=true実装済み
deckCode→cardList一括変換（register_metaの入力材料作成用）
内部処理pokeka2 Service Binding呼び出し→8分類マッピング→ポケモンはsetInfo変換、トレーナーズ／エネはsetCode＋名前完全一致でcardId照合
出力cardList（8分類構造）＋unmappedCategories
備考詳細な内部分岐は「処理フロー図：resolve_cardlist」参照
④ レギュレーション判定
?check_set=true実装済み
新弾差分の確認
?recheck_all=true実装済み
登録済み全デッキのレギュ違反を一括再判定する
実行タイミング年1回のレギュ改定タイミングのみ（決定事項）
備考自動削除・自動修正はせず、違反マーク表示のみ。recheck_mine（provisional再照合）とは目的が違うので混同注意
（register_meta/register_mine/update_mine内で自動実行）独立エンドポイントではない
register時のレギュ検証はcardList変更を伴う保存処理の内部で実行される
備考provisional（cardId未確定）のカードは判定対象から除外。confirmedのカードのみ判定する（2026-07-20仕様修正）
⑤ 初期化・同期
?init=true実装済み
スタンダード合法28セット分のカードデータを初回一括ロードする
注意初回一括ロード専用。新セットの差分追加には使えない（sync:progressが巻き戻るため、実コード確認で判明・2026-07-20）
?sync=true実装済み
初回一括ロードの続き・再開用
注意?init=true同様、差分追加用途には使えない
（仮称）?add_set=true構想のみ・今後検討
新セット1件だけをKVへ差分追加するための専用エンドポイント（案）
対象SVシリーズ本弾のみ（いずれTCGdexに載るもの）。日本限定パックはTCGdex非対応のため構造的に対象外
